##### 背景技术扫盲——oauth2+opaque token

oauth2的四种授权模式+1种刷新机制

| 四大类型 | 项目使用 | 用途                | 补充                            |
| -------- | -------- | ------------------- | ------------------------------- |
| 密码     | √        | 自家前端            |                                 |
| 授权码   | √        | 三方（单点登录SSO） |                                 |
| 客户端   | √        | 应用触发            | MQ消费/ 批处理（User 变为 App） |
| 简化前端 | ×        | 简化模式            | 过时                            |
|          |          |                     |                                 |
| 刷新机制 | ×        |                     | Token续签                       |

区分不严格：

- 如果是账户密码登录的user下单了，微服务链路就是**密码模式**
- 如果是微信授权登录后下单，那就是**授权码模式**
- 如果超时取消订单，那就是**客户端模式**（超时系统自动取消）

（实际上都是完全相同的代码在跑）



##### 项目鉴权

- 登录用到：
  - 密码模式：
  - 微信授权（授权码）：
  - 

- 微服务之间：

  - A微服务Header 透传token  -->  B微服务本地验签（本项目不使用这种方法） ； 
  - A微服务Header 登录信息json打包放header  -->  B微服务reset登录信息为A传用户；
    - 登录信息指的是：SecurityFrameworkUtils.getLoginUser()
    - reset的意思是：SecurityFrameworkUtils.setLoginUser(loginUser, request)
    - SecurityFrameworkUtils原理：通过 **ThreadLocal** 绑定鉴权信息到请求 线程 （only）；
      - 通过SpringSecurity 操作存入ThreadLocal & SecurityContextHolder上下文
    - SecurityFrameworkUtils.getLoginUser()随时都可调用，里面的user生命周期是：
      - 请求结束就销毁；
      - 每次请求进来，都会被拦截器（TokenAuthenticationFilter）塞入ThreadLocal，拦截器会走之前说的if json then 直接用 else if token then查redis/db
    - 注意：@Async之类的子线程是拿不到的；需要手动折腾传递；

  

-  前端后端之间： 

  - 前端header透传token  -->  gateway --1min缓存未中-> 调用system.checkToken--> redis --未中--> DB；
  - 前端header透传token -->  gateway本地验签（本项目不使用）

- gateway统一鉴权
  - A微服务Header 透传 .  B微服务本地验签 （本项目不用）
  - gateway调用system.checkToken





##### 项目总结

- 采用了 Gateway 统一鉴权 + Opaque Token 的微服务安全架构
  亮点在于：
  - 通过 Gateway 本地缓存 解决了 Opaque Token 频繁查库的性能瓶颈
  - 通过 Header 透传用户上下文 实现了微服务间的无状态高效通信
