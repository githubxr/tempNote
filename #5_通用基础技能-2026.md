##### 基础 **Filter / FilterChain** 要点

-  Filter：自行继承重写配置过滤操作
   -  配置方式1：
      -  通过具体的【@WebFilter("/admin/*") 】类似的写法，实现功能性拦截；
      -  通过【@Order(顺序整型)】控制先后顺序 （or xml配置顺序）
   -  配置方式2：
      -  通过springBoot的@Configuration + @Bean的形式 配置实例
      -  路由和顺序：通过 Bean方法返回的 return xx.pattern匹配路由.setOrder配置先后顺序...
   -  配置方法3（微服务）：
      -  spring.cloud.gateway.routes.【id/ uri/ predicates/ filter(-请求头过滤...；-裁剪路径...)】
   -  备注（不一定只用某一种，例如3可以和2搭配，3配路由，2配具体定制的鉴权/权限等骚操作）
-  FilterChain：无需继承重写，Filter的执行方法中的参数，拿出来拦截 or 可触发下一个Filter执行
-  OncePerRequestFilter（http请求的filter基本都用这个）:
   -  含义：一个请求只会触发一次
   -  意义：避免可能的（重定向/转发）重复触发Filter

##### SQL进阶

- 联表提前过滤的注意事项：

- 压缩：

  ```
  INNER JOIN：
      随便放
  
  LEFT JOIN：
      左表条件放WHERE
      右表条件放ON
  
  RIGHT JOIN：
      左表条件放ON
      右表条件放WHERE
  ```

- 进阶：

```
（因为：on过滤 => 联表 => Left/Right补Null => Where 顺序）
- inner join 时，可以同时把左表右表过滤条件都写on内（推荐： 在联表前过滤）

- left join 时
  - 右表过滤条件：可无脑写On内，也应该写On内
    - 补充：若右表条件不写On内，会导致左连接变成类似‘内连接’
  - but 左表过滤条件：需特别注意，左表条件在On，只是 ‘筛选和右表连接的左表行’
    - 因左连接特性，筛选下来的，未和右表匹配的，也会保留。
    - 因此如果试图在左连接的on中过滤左表，意图会落空；

- right join时：
  （和左连接差不多反过来）
  - 右表过滤条件：如果试图在On过滤右表，意图会落空
  - 左表过滤条件：可无脑写On
    - 补充：若写Where中，会导致右连接变成类似‘内连接’
```

- 普通二叉树

  - 三种遍历方式：
    - 前入：根左右
    - 后入：左右根
    - 中出：左根右
  - 完全二叉：最后一层可不满（必须从左到右连续，只有右边可以空缺）
    - 例如底层有abcd，其中可以有a,ab,abc三种形式都属于完全，而abcd都有，则是满二叉树
  - *满二叉：所有层都塞满，最后一层塞满（简单/现实几乎不用）——次要*
  - **二叉搜索树**：有序的树
    - 唯一规则：allLeftChildNode < parentNode < allRightChildNode
    - 唯一作用：可二分式查找
    - 不做db，因为：数据是连续的12345，就会退化为链表（root->childLvel1->childLevel2->...）
      - 因此可避免退化的 **平衡树** 就有意义，例如 **红黑树**

- B+树：

  - 非二叉树！是N叉树
    - 1个叶子有多个line，and 同时数据结构也是多x树，每个根都可以有N个叶子/子节点
  - 生长规律：
    - 普通二叉树：向下‘开枝散叶’（植物生长）
    - B+树：向右分裂/向上长根（盖房子）
  - 特殊性
    - 数据存储：
      - only 叶子节点数据，叶子节点单位为’页‘，可存多行数据
      - 页满了才会’长其它节点‘（触发分裂 / then 自动平衡调整结构）
      - 叶子（页）之间有指针，连起来为 **链表**
    - 非叶子节点
      - 用来存索引key + 叶子指针
  - 索引
    - 聚簇索引，就是索引字段值在非叶子节点作为数据节点（数据节点存整行数据）的父节点；
    - 非聚簇索引，就是独立的B+树，叶子节点存完整行数据，非叶子节点存非聚簇索引；
      -  回表问题：
        - 因此，得拿到叶子节点的聚簇索引去‘主树’找...
        - 若‘覆盖索引’了（每个select字段都有非聚簇索引（独立的B+树）），那就不用那叶子查到的主键值，去‘主树’找了。。。
  - 面试题
    - 高并发写导致‘页分裂’抖动性能怎么办
      - 主键使用自增ID / Snowflake算法，以每次在‘最右末尾’追加，规整分裂右边，避免‘中间插队’；

- B树：

  - 非叶子也可以存数据的B+数，因为非叶子存数据挤占索引空间导致树高度比B+高而被B+替代

##### 分库分表核心

- 垂直拆分
  - 分库：分到另一个主机
  - 分表（拆列）：应对场景是’by bussiness‘，如部分字段体积较大 访问频率较低，db每次查询都得查出来才过滤，这个会降低效率，因此把这些列拆出去提高查询效率；
- 水平拆分
  - 分表：表存在不同db
  - 拆行：应对场景是 解决数据行数过于庞大的问题，把行拆分到多个DB主机，以让IO效率更高；有多种拆的方式；
    - 按id（范围路由/hash取模）之类的取模
- 同样的，更好扩展能力同时带来查询/事务和运维复杂度的提升，符合’奥卡姆剃刀原则‘

##### 分布式锁要点：**看门🐕**

- 本质：web程序中的Redission实例中的续期任务机制

- 作用：负责自动续期🔒，避免业务执行时间超过锁TTL导致提前释放

- 生效规则：只要 不传leaseTime都会自动加看🚪🐕
  - 最长等待时间为：Redisson 全局配置：`lockWatchdogTimeout`（默认 30s）
- 用法：
  - rLock.lock()：
  - rLock.tryLock()：推荐
    - waitTime：获取最多等多久🔒
    - leaseTime：固定释放🔒时间（会关看门狗）
    - 推荐写法：tryLock(waitTime, timeUnit)

- 结束要点
  - 别忘了.unlock() ：触发‘杀🐕’
    - 删锁 + 停止续期
  - or被动的：jvm挂了，RedissonClient也挂，🐕也挂，因此不再续期
